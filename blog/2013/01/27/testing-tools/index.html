Testing Tools

Jan 27th, 2013

Automated testing is becoming a necessity rather than just a nice to have. It allows us to have far more confidence in the code we’re about to ship so we have less late night fixes and embarrassing apologies.

We’re becoming more reliant on automated tests as our project evolves:

Growing code base

Our team has been growing rapidly over the last few months and in doing so commits to the code base has increased too. We’re rapidly deploying all of the BBC 28 World Service sites which puts more pressure on the manual testers.

This obviously doesn’t scale and leads to more time being spent at the end of the cycle just for testing halting the whole process.

More regular deploys

We’re currently running on a two week deploy cycle, mainly due the the fact the deployment process at the BBC is pretty manual and completely owned by Ops (stage and live).

As with most dev teams we’re pushing for continuous deployment to help us make more frequent less risky changes to the code base. But a one click deploy is looking like a few years own the line in the mean time we’re looking at weekly deploys.

Device explosion

Thousands of different devices hit our site, some with crazy browsers and manufacturers that love to tinker with the stock experience. We regularly scan our stats to make sure we have most popular devices at hand - we don’t feel on-line solutions for device testing is worthwhile.

Responsive design

We’re big fans of responsive design (duh), but the big challenge with RWD is managing the complexity of the front end code.
